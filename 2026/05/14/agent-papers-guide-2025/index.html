AI Agent 必读论文导读：三篇搞懂 Agent 设计核心

发表于2026-05-14|更新于2026-05-14|AI Agent
|浏览量:

Agent 领域论文满天飞，但真正值得细读的不多。本文精选三篇分别来自 Anthropic、OpenAI、Google DeepMind 的核心论文，从设计哲学、工程实践、科学原理三个维度，帮你建立对 AI Agent 的完整认知框架。

一、设计哲学：Building Effective Agents

作者: Anthropic
链接: anthropic.com/research/building-effective-agents

这篇在讲什么

Anthropic 把 Agent 的构建拆成了一个清晰的递进模型，帮助开发者判断「我到底需不需要用 Agent」。

核心架构：三层递进

第一层：Augmented LLM（增强型 LLM）

最基础的形式。给 LLM 加上检索、工具调用、记忆等能力，但它本身不自主决策，只是被动响应。

第二层：Workflow（工作流）

开发者预先定义好流程图，LLM 按照固定的编排逻辑执行。比如：

  • 先分类用户输入 → 再调用对应工具 → 最后格式化输出
  • Prompt Chaining：第一个 LLM 的输出作为第二个 LLM 的输入
  • Routing：根据输入类型分发到不同的处理分支

第三层：Agent（智能体）

LLM 自己决定下一步做什么。没有预定义流程，模型根据当前状态自主选择工具和行动路径。

关键洞察

1. 不要过度设计

大多数场景用 Workflow 就够了，只有在需要模型自主判断和动态决策时才上 Agent。

Anthropic 的建议是：从最简单的方案开始，验证不够用再升级。很多开发者上来就搞复杂的 Agent 框架，实际上 Prompt Chaining 就能解决。

2. 工具设计 > Prompt 优化

在构建 SWE-bench Agent 时，Anthropic 发现花在优化工具上的时间比优化 prompt 更多，而且效果更好。原因很简单：

  • Prompt 只影响模型的「理解」
  • 工具决定了模型能「做什么」以及「做得多好」
  • 好的工具定义 = 好的 Agent 表现

不需要复杂的框架，关键在于：给模型清晰的工具描述、合理的退出条件、以及足够的上下文。

为什么必读

这篇文章最大的价值不是教你写代码，而是帮你建立 Agent 设计的思维模型。它回答了一个根本问题：「我的场景到底需要多复杂的 Agent？」答案通常是：比你以为的简单得多。

二、工程实践：Harness Engineering

作者: OpenAI
链接: openai.com/index/harness-engineering

这篇在讲什么

OpenAI 内部用 Codex（他们的编码 Agent）以 0 行手写代码 构建了一个完整产品，效率提升约 10 倍。这篇文章记录了他们在这个过程中学到的一切。

什么是 Harness Engineering

传统软件工程的核心是 写代码。但在 Agent-first 的世界里，人类的角色变了：

传统开发 Agent-first 开发
人类写代码 Agent 写代码
人类设计架构 人类设计环境
人类调试 bug 人类构建反馈循环
人类做 code review 人类定义验收标准

Harness Engineering = 设计让 Agent 高效工作的「环境」（Harness）

三大核心原则

1. 环境设计 > 代码编写

Agent 需要的不是一个好的 prompt，而是一个好的工作环境：

  • 目录结构要清晰，让 Agent 能理解项目组织
  • 配置文件要完整，让 Agent 能直接运行和测试
  • 文档要准确，让 Agent 能查阅上下文

你不是在写代码给机器执行，你是在设计一个「车间」让 Agent 在里面高效工作。

2. 反馈循环 > 一次性 prompt

Agent 不是调用一次就完事的。关键是构建闭环：

OpenAI 发现，最好的结果不是来自完美的 prompt，而是来自快速的迭代反馈。Agent 跑偏了没关系，关键是能快速纠正。

3. 可观测性 > 事后调试

Agent 的行为是不可预测的，所以你需要：

  • 完整的执行日志（Agent 做了什么、为什么这么做）
  • 中间状态快照（每一步的结果是什么）
  • 人工介入点（什么时候需要人类确认）

实际案例

OpenAI 用 Codex 构建的那个产品，整个开发过程是这样的：

  1. 人类描述需求和意图
  2. Codex 生成代码实现
  3. 人类评估结果，给出反馈
  4. Codex 修改和迭代
  5. 重复 2-4 直到满意

最终效果：开发时间缩短到原来的 1/10，而且代码质量不比手写差。

为什么必读

这篇文章改变了我对「软件工程」的认知。当 Agent 能写代码时，人类的核心价值不再是写代码本身，而是 设计让 Agent 高效工作的环境。这种思维转变对每个开发者都很重要。

三、科学原理：Towards a Science of Scaling Agent Systems

作者: Google Research
链接: research.google/blog/towards-a-science-of-scaling-agent-systems

这篇在讲什么

「多 Agent 一定比单 Agent 好吗？」——Google 用 180 种 Agent 配置的系统实验，首次给出了量化的答案。

实验设计

Google 在 4 个不同的 Agent 基准上测试了各种配置：

  • 信息检索: 从多个网站抓取和综合信息
  • Minecraft 规划: 在游戏环境中完成复杂任务
  • 业务任务: 处理邮件、安排会议、使用项目管理工具
  • 金融分析: 金融数据处理和决策

配置变量包括：Agent 数量、任务分配策略、通信方式等。

核心发现

发现一：多 Agent 在可并行任务上效果显著

当任务可以被拆分成独立的子任务并行执行时，多 Agent 的效果远超单 Agent。比如「同时搜索 5 个不同来源的信息」——每个 Agent 独立工作，互不干扰。

发现二：多 Agent 在顺序任务上反而退化

当任务是严格顺序依赖的（A 的输出是 B 的输入），多 Agent 的效果反而不如单 Agent。原因是：

  • Agent 间的信息传递有损耗
  • 每次传递都可能引入误解
  • 协调成本抵消了并行收益

发现三：架构选择可以被量化预测

Google 构建了一个预测模型，只需分析任务的特征（可并行度、依赖深度、复杂度），就能预测最优的 Agent 架构。准确率达到 87%。

关键公式

Google 提出了一个直观的评估框架：

实践启示

  1. 先分析任务结构，再决定用几个 Agent
  2. 能并行就拆分，不能并行就保持单 Agent
  3. Agent 数量有最优解，不是越多越好
  4. 通信频率是隐形杀手，Agent 间通信太多会拖垮性能

为什么必读

大多数 Agent 论文都在讲「怎么做」，这篇在讲「什么时候做、做多少」。它把 Agent 系统设计从「拍脑袋」变成了「有据可依」。如果你在纠结「该用单 Agent 还是多 Agent」，这篇直接给你答案。

三篇论文的认知地图

推荐阅读顺序:

  1. 先读 Anthropic → 建立设计思维
  2. 再读 OpenAI → 理解工程实践
  3. 最后读 Google → 掌握决策框架

Agent 不是银弹，但它是当前 AI 应用最重要的范式之一。这三篇论文加起来不到 2 小时能读完，但能帮你省下几十小时的试错时间。
